riverside-county-auditor-controller-mock-chatbot

What this is

An internal admin sandbox for the Riverside County Auditor-Controller chatbot. The widget on the right is the same one production claimants see embedded on auditorcontroller.org — it talks to the same WebSocket, the same Bedrock agent, and the same backend.

How to interact with the bot

  • Give a name — try "My name is Carey Ministries" or just "Carey Ministries". The bot fuzzy-matches against the demo dataset.
  • Address quiz — after a name match, you'll see 4 street-name buttons. Pick the right one (the bot won't tell you which) to continue.
  • House number — enter the street number and click Verify.
  • Refunds reveal — once verified, the bot lists every refund on file with amount + deadline + a personalized claim-portal link.
  • Live-agent handoff — type "I'd like to talk to a person" to exercise the reference-number flow. The handoff appears on the admin dashboard's Chat handoffs page.
  • FAQ questions — try "How long does it take to process a claim?" or "Is there a deadline?".

The two buttons in the chat header

  • ↻ Restart — fully resets the conversation. Clears the transcript, drops your verification status, and starts a new session ID. Use this between test scenarios so prior verification can't leak across.
  • × Close — does nothing useful on this page. The widget normally closes back to a chat bubble; on the tester the bubble is hidden so closing leaves a blank page. Refresh to bring the chat back.

What persists across page loads

  • In this tab: the session ID stays in sessionStorage, so reloading the page (without hitting ↻) keeps your conversation, identity verification, and lockout state intact on the server.
  • Closing the tab: the session ID is cleared. A new tab gets a brand-new session.
  • Server-side: conversation transcripts and verification state live in DynamoDB and persist regardless of what your browser does. Restart wipes the client-side reference but the old transcript is still on the admin dashboard.
  • Lockouts: 5 wrong street/number guesses lock the session for 1 hour. Lockouts are server-side and tied to the submission, so opening a new tab won't bypass them.

Things to watch for during testing

  • Same name twice should produce identical 4 street options — quiz decoys are deterministic per claimant.
  • The bot should never reveal refund details before identity is verified.
  • Wrong-street guesses should fail gracefully without telling you which was correct.
  • The portal link in the refunds reply should land on the v3 claimant portal (CloudFront URL, not the v2 stack).
